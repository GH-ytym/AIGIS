# 武汉市GIS智能助手技术设计文档（LangChain + MiniMax + Leaflet + 腾讯MCP）

## 1. 文档目的

本技术文档用于指导后续从零开发一个“仅服务武汉市范围”的GIS智能助手系统。系统定位是面向真实业务场景的城市级空间智能应用，主要解决自然语言问询、地点检索、路径规划、周边发现、时空决策支持等问题。文档阶段不写业务代码，仅明确边界、架构、流程、模块职责、接口约束、开发节奏和最小文件骨架，以便后续开发团队直接进入实现阶段。

本项目的核心技术组合固定为四部分：
1. LangChain 作为智能体编排框架。
2. MiniMax 作为主语言模型服务。
3. 腾讯地图MCP作为地图与位置能力接入层。
4. Leaflet 作为前端地图可视化与交互载体。

本项目采用“先架构后编码、先约束后扩展、先武汉再外延”的策略，确保第一版可以快速上线并具备稳定演进能力。

## 2. 需求定义与边界

### 2.1 业务目标

系统目标不是做通用聊天机器人，而是做“能在地图上行动”的城市智能助手。用户输入自然语言后，系统应当完成意图理解、任务拆解、工具调用、结果融合和地图渲染，并最终返回可操作的结果。

### 2.2 关键要求

1. 服务范围严格限定在武汉市域内。
2. 允许在开发阶段暴露Key，降低接入门槛。
3. 使用腾讯MCP作为核心地理能力入口。
4. 文档先行，不写实现代码。
5. 建立少量文件的清晰骨架，降低初始复杂度。

### 2.3 非目标

1. 当前阶段不实现多城市切换。
2. 当前阶段不实现复杂权限系统。
3. 当前阶段不实现高并发优化。
4. 当前阶段不实现数据库持久化策略。

## 3. 对你提出流程理解的确认

你的理解是正确的，而且是本方案的主路径：

用户输入自然语言后，后端先调用LLM做任务拆解和规划，再由LLM基于腾讯MCP工具能力决定调用顺序与参数，最后将执行结果整合为用户回复与地图动作。

在工程上，需要补充三个关键点：
1. 不是让LLM直接无限制调用工具，而是放在“受控编排框架”里运行。
2. 工具调用前后要做参数校验和区域约束，确保始终不出武汉。
3. 前端只消费标准化地图动作，不直接耦合具体MCP工具名字。

## 4. 武汉市域约束策略

武汉限定是本系统最重要的规则，不应仅靠提示词，应采用“多层约束”。

### 4.1 提示层约束

系统提示词明确声明：所有检索、路径、推荐均以武汉市为默认和唯一有效范围。若用户输入外地地名，优先进行澄清或拒绝执行外域任务。

### 4.2 参数层约束

所有进入MCP调用前的参数必须经过城市边界规范化：
1. 地名参数补全为武汉行政上下文。
2. 坐标参数进行边界校验。
3. 关键字检索默认加武汉范围条件。

### 4.3 结果层约束

即使上游接口返回了非武汉结果，也必须在结果融合阶段再次过滤，仅保留武汉范围内对象，避免“跨城污染”。

### 4.4 交互层约束

当前版本不提供城市选择器，不暴露切换城市能力。若用户强制要求外地信息，系统返回能力边界提示。

## 5. 总体技术架构

系统采用四层分离架构：接入层、编排层、能力层、表现层。

### 5.1 接入层

负责接收前端请求、维护会话上下文、返回结构化响应。该层不负责具体工具细节，仅负责调用编排层。

### 5.2 编排层

由LangChain驱动，包含如下职责：
1. 解析用户意图。
2. 生成任务计划。
3. 选择工具序列。
4. 执行顺序控制。
5. 处理重试与降级。
6. 产出最终文本和地图动作。

### 5.3 能力层

以腾讯MCP为核心能力入口，统一封装为“工具网关”。能力层仅处理与工具有关的问题，如参数映射、调用超时、错误翻译、结果标准化。

### 5.4 表现层

Leaflet负责地图交互。前端只做三件事：
1. 提交自然语言。
2. 渲染地图动作。
3. 展示助手解释与步骤反馈。

## 6. 模块职责设计

### 6.1 Agent编排模块

Agent编排模块建议职责如下：
1. 接收用户输入与会话历史。
2. 产出任务计划对象。
3. 逐步执行任务。
4. 记录每一步工具调用日志。
5. 汇总最终答复和地图动作。

任务计划建议包含：任务目标、子任务列表、所需工具、参数草案、成功判定条件、失败回退策略。

### 6.2 MCP网关模块

MCP网关模块是系统可靠性的关键，建议统一封装以下逻辑：
1. 工具发现。
2. 工具白名单。
3. 参数Schema校验。
4. 超时控制。
5. 错误码分类。
6. 返回文本结构解析。
7. 结果标准化。

同时提供“工具健康状态缓存”，便于前端展示当前能力可用性。

### 6.3 武汉约束模块

该模块可视为独立策略组件，职责包括：
1. 地名归一化到武汉上下文。
2. 坐标武汉边界判定。
3. 外域结果剔除。
4. 违规请求响应模板。

### 6.4 地图动作转换模块

后端应将工具结果转换成统一动作协议，例如：
1. 设置中心点。
2. 添加标记点。
3. 绘制路线。
4. 绘制服务范围。
5. 更新结果列表。
6. 输出解释文案。

前端只解释动作，不关心其来源工具。

## 7. 核心执行流程

### 7.1 主流程

1. 用户在前端输入自然语言。
2. 接入层建立请求上下文。
3. 编排层调用MiniMax生成任务计划。
4. 计划经武汉策略模块校验。
5. 编排层驱动MCP网关按顺序执行工具。
6. 工具结果进入标准化与融合。
7. 生成文本答复和地图动作列表。
8. 前端Leaflet渲染并反馈给用户。

### 7.2 失败与降级流程

1. 工具超时时进行一次重试。
2. 工具不可用时切换备选工具。
3. 无备选时返回明确可解释失败原因。
4. 关键步骤失败仍要返回部分可用结果，避免整单失败。

### 7.3 会话流程

建议保留短会话上下文，避免模型遗忘用户地点偏好与任务目标。会话数据只保存必要信息，不保存敏感个人数据。

## 8. 工具编排策略

### 8.1 工具选择原则

1. 先低成本检索，再高成本规划。
2. 先结构化结果，再叙述化结果。
3. 先确认位置，再做路线与推荐。

### 8.2 参数生成原则

1. 参数由LLM提出草案。
2. 网关进行强校验和补全。
3. 武汉策略进行范围约束。
4. 不符合规则的参数直接拒绝执行。

### 8.3 结果融合原则

1. 结果去重。
2. 结果按相关性排序。
3. 结果附带来源工具与可信度标识。
4. 结果必须可映射到Leaflet动作。

## 9. Key暴露策略（按当前要求）

本项目当前阶段允许暴露Key，但必须明确其“阶段性、可替换、受限用途”的性质。

### 9.1 当前策略

1. 在配置样例中直接给出MCP Key。
2. 默认仅用于开发与联调环境。
3. 默认只开放武汉场景所需接口。

### 9.2 风险提示

1. 暴露Key会带来额度消耗和滥用风险。
2. 一旦上线公网，建议切换为环境变量注入和服务端代理隐藏。
3. 建议预留快速换Key机制。

## 10. 观测与诊断设计

为了避免“调用成功但结果不可用”的错觉，必须有可观测层。

建议至少记录：
1. 请求ID。
2. 用户输入。
3. 任务计划。
4. 工具调用顺序。
5. 每次调用参数。
6. 每次调用结果摘要。
7. 错误分类。
8. 最终地图动作数量。

前端建议展示简版调试信息：当前步骤、当前工具、当前状态。

## 11. 前端交互设计（Leaflet）

### 11.1 交互目标

让用户看到“助手正在做事”，而不是只看到静态答案。建议采用“消息气泡 + 地图渐进渲染”的交互方式。

### 11.2 必要界面元素

1. 自然语言输入框。
2. 对话输出区。
3. 地图主画布。
4. 结果列表面板。
5. 状态提示条。

### 11.3 地图行为规范

1. 首次定位到武汉中心。
2. 每次任务只保留一组主结果图层。
3. 新任务触发时清理旧任务临时图层。
4. 点击结果卡片与地图标记双向联动。

## 13. 接口与数据协议建议

### 13.1 统一请求

建议保留一个核心问答接口，输入为自然语言和会话标识。

### 13.2 统一响应

响应建议包含四部分：
1. 文本答复。
2. 地图动作数组。
3. 工具执行摘要。

2. add_markers

## 14. 开发阶段计划

## 15. 验收标准

### 15.1 功能验收

1. 用户输入武汉相关查询后，可稳定返回地图可视结果。
2. 工具调用顺序可追踪。
3. 结果限制在武汉域内。
4. 错误信息可解释。

### 15.2 质量验收

1. 失败任务不会导致服务崩溃。
2. 关键请求具备日志闭环。
3. 前端不会因单个动作失败而白屏。

### 15.3 文档验收

1. 文档与目录结构一致。
2. 文档覆盖约束、流程、模块、协议、计划。
3. 开发人员可直接基于文档开工。

## 16. 风险与对策

### 16.1 MCP工具返回形态不稳定

对策：网关统一做文本结构化提取和结果校验，不将原始文本直接透传业务层。

### 16.2 单一工具受配额或授权限制

对策：建立工具可用性探测和替代链路，编排层支持动态降级。

### 16.3 模型规划误差导致工具参数不合理

对策：参数强校验与规则修正必须在网关层执行，不满足规则不执行。

### 16.4 Key暴露导致额度风险

对策：当前阶段接受风险，但必须预留一键切换Key和额度告警机制。

## 17. 结论

本方案确认了你的核心理解并将其工程化：
1. 自然语言进入后由LLM拆解任务。
2. 由LLM在受控框架内决定MCP工具调用顺序和参数。
3. 后端对调用进行约束、校验、降级和结果标准化。
4. 前端Leaflet只渲染动作，形成可视化智能GIS助手。

在当前阶段，这是一套可执行、可迭代、可扩展的起步方案。下一步即可进入“按骨架逐步填充代码”的开发周期。
